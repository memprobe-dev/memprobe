How to read your ELF memory map

When you compile firmware for an embedded target, the linker produces an ELF (Executable and Linkable Format) file. An ELF file is more than a flat binary — it contains rich metadata about every function, variable, and section in your program. memprobe reads that metadata and turns it into an interactive memory breakdown.

What is a memory map?

A memory map describes how your program's code and data are laid out in the target's address space. Each named region in the map corresponds to a physical memory area on the chip — most commonly Flash (for code and read-only data) and RAM (for runtime data and the stack).

The linker script for your target defines these regions. For example, an STM32F4 might have:

  • FLASH — 1 MB starting at address 0x08000000
  • RAM — 192 KB starting at address 0x20000000
  • CCMRAM — 64 KB of tightly-coupled RAM at 0x10000000

Flash usage vs. RAM usage

One of the most important distinctions in embedded firmware is the difference between Flash usage and RAM usage:

  • Flash usage is how much non-volatile storage your firmware occupies. This determines whether your binary fits on the chip and how much space is left for future updates.
  • RAM usage is how much working memory your program needs at runtime. This includes global and static variables, the heap, and the stack. If RAM overflows, your program crashes — often in subtle, hard-to-debug ways.

These two numbers can be very different. A function full of constants may use a lot of Flash but almost no RAM. A large lookup table stored in Flash with a const qualifier costs Flash, not RAM.

Reading the memprobe output

When you analyze a file with memprobe, the top-level numbers — Flash and RAM — are derived from the sections in your ELF and the regions defined in your linker script. Below that:

  • Sections — The raw ELF sections (.text, .rodata, .bss, etc.) sorted by size. These are the building blocks of your memory map.
  • Symbols — Individual functions and variables within those sections. The symbol table lets you pinpoint exactly which function is consuming the most Flash or which global variable is eating your RAM.
  • Treemap — A proportional visual of section sizes. Larger rectangles mean more memory.
  • Regions — A fill-percentage bar for each memory region defined in your linker script, so you can see at a glance how close you are to the limit.

Tip: Sort the symbol table by size and look at the top 10 entries. In most firmware projects, a small number of symbols account for the majority of Flash or RAM usage. Fixing those first gives the biggest return.

Linker map files (.map)

If you do not have the ELF file, memprobe also accepts .map files generated by the linker (GCC's -Map flag, or IAR's map output). Map files contain similar section and symbol information in a text format. They are less detailed than ELF files — binary info like architecture and entry point will be missing — but the memory breakdown is still accurate.

Memory section types: .text, .rodata, .bss, .data, and more

ELF sections are named regions within the binary, each with a specific purpose. Understanding what goes into each section tells you where your memory is being spent.

Flash sections (read from non-volatile storage)

SectionContainsMemory type
.text Compiled machine code — every function in your program. This is almost always the largest section in a firmware binary. Flash
.rodata Read-only data: string literals, const arrays, lookup tables, and other compile-time constants. Because it never changes, the linker places it in Flash rather than RAM. Flash
.data (LMA) The initial values for initialized global and static variables. At startup, the C runtime copies this from Flash into RAM. The Flash copy is sometimes called the Load Memory Address (LMA) copy. Flash (stored), RAM (at runtime)
.isr_vector / .vectors The interrupt vector table — a fixed array of function pointers that the CPU jumps to when an interrupt fires. Located at the start of Flash on most Cortex-M devices. Flash
.ARM.extab / .ARM.exidx ARM exception-handling tables used for stack unwinding (C++ exceptions or debugger backtraces). Often present even in C-only projects when using GCC's ARM target. Flash

RAM sections (used at runtime)

SectionContainsNotes
.data Initialized global and static variables — those with a non-zero initial value. For example, int x = 5; at file scope. Counts against both Flash (for the initial values) and RAM (for the runtime copy).
.bss Uninitialized (or zero-initialized) global and static variables. For example, int x; or static uint8_t buf[1024];. The C runtime zeroes this region at startup — no Flash copy needed. Only costs RAM, not Flash. Often the largest RAM section.
.heap Dynamic memory managed by malloc/free. The linker reserves a fixed block; actual usage depends on runtime behavior. Avoid on resource-constrained targets. Fragmentation is a common failure mode.
.stack The call stack — local variables and return addresses for every active function call. The linker typically reserves a fixed size; overflow causes a hard fault. Increase stack size if you see unexplained hard faults or corrupted variables.
.noinit Variables that must survive a soft reset — not zeroed by the startup code. Useful for storing crash diagnostics or boot counters across resets. Requires explicit linker script support.

Common patterns and what to look for

  • Large .rodata: Usually caused by string tables, font bitmaps, audio samples, or large const arrays. These are fine in Flash but check if any are accidentally in RAM.
  • Large .bss: Often caused by large static buffers or global arrays. Consider whether they can be allocated on the stack (local variables) or replaced with smaller ring buffers.
  • Significant .data: Every byte here costs Flash twice — once for the stored initial value and once for the RAM copy at runtime. Prefer zero-initialization (.bss) or const data (.rodata) where possible.
  • Many small symbols in .text: May indicate excessive inlining or template instantiation. Check compiler flags — -Os optimizes for size, while -O2/-O3 optimize for speed at the cost of code size.

Quick rule of thumb: Flash usage = .text + .rodata + .data (LMA). RAM usage = .data (VMA) + .bss + .heap + .stack.
